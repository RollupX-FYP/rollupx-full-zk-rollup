The core of this project is the design, implementation, and rigorous benchmarking of a modular ZK-Rollup framework. This framework will be deployed on an Ethereum testnet to ensure that our findings are grounded in a realistic network environment. The initial implementation will support a focused set of operations—namely, simple token transfers, deposits, and withdrawals—which serve as the foundation for our performance analysis. A key feature of the framework is its configurability, which will allow us to systematically vary experimental parameters such as batch size, sequencer scheduling policies, and data availability modes. Furthermore, the architecture will be intentionally designed to be extendable, thereby allowing for future evolution and the integration of more complex functionalities.

\subsection{Prototype Design and Implementation}
\subsubsection{System Architecture}
To ensure a clear separation of concerns and to facilitate modular experimentation, the prototype will be structured into a five-layer architecture. Each layer will have a distinct responsibility, from transaction submission at the top to data availability at the bottom. This design choice is critical for isolating variables during benchmarking and for allowing future extensions to the framework.

The components of the prototype are as follows:
\begin{itemize}
    \item \textbf {User Layer:}
    The user layer is the entry point for end-users to interact with the system.
    \begin{itemize}
        \item {Wallets / Clients:} End-users initiate deposits, transfers, and withdrawals by signing transactions.
        \item {Synthetic Workload Generator:} Provides benchmark workloads for performance evaluation.
    \end{itemize}
    
    \item \textbf{Off-Chain Components:}
    These components handle the core logic of the rollup, including transaction processing, state management, and proof generation.
    
    \begin{itemize}
        \item \textbf{Sequencer:}
        The sequencer is responsible for collecting and ordering transactions.
        \begin{itemize}
            \item {Transaction Pool:} Collects and stores pending user transactions.
            \item {Batching Engine:} Groups transactions into batches based on configurable policies (e.g., batch size, timeout, or priority).
            \item {Scheduler (optional):} Supports custom policies for sequencing order (FIFO, priority-based).
        \end{itemize}
        
        \item \textbf{Executor:}
        The executor processes the transactions and updates the off-chain state.
        \begin{itemize}
            \item {State Manager:} Maintains the off-chain state (i.e., account balances and nonces) using a Sparse Merkle Tree (SMT). We plan to adapt an existing JavaScript library such as \texttt{js-merkle} or the implementation used by Polygon's zkEVM for this purpose.
            \item {State Transition Engine:} Applies transactions to update the state and compute new state roots.
            \item {Block Constructor:} Forms transaction blocks with corresponding state updates.
        \end{itemize}
        
        \item \textbf{Prover:}
        The prover generates cryptographic proofs to ensure the correctness of state transitions.
        \begin{itemize}
            \item {Circuit Backend:} zkSNARK/STARK circuits verify state transitions ($\text{OldRoot} \rightarrow \text{NewRoot}$) \cite{sheybani2025zero}.
            \item {Default Backend:} Circom + Groth16/Plonk for rapid prototyping \cite{iden3_circom, sui_groth16, polygon_plonky2}.
            \item {Optional Backends:} Plonky2 or Halo2 for performance comparisons \cite{polygon_plonky2, zcash_halo2}.
            \item {Proof Generator:} Produces validity proofs ensuring correctness of state transitions.
        \end{itemize}
        
        \item \textbf{Submitter:}
        The submitter packages and sends the rollup data to the Ethereum mainnet.
        \begin{itemize}
            \item {Batch Packager:} Packages transactions, state roots, and validity proofs.
            \item {On-chain Broadcaster:} Submits canonical state roots, ZK proofs, and DA commitments to Ethereum.
            \item {Data Availability Connector:} Chooses DA mode (calldata, compression, or external DA).
        \end{itemize}
    \end{itemize}
    
    \item \textbf{On-Chain Components}
    These are the smart contracts deployed on the Ethereum mainnet that serve as the anchor for the rollup.
    \begin{itemize}
        \item \textbf{RollupBridge.sol}
            \begin{itemize}
                \item Stores canonical state roots.
                \item Manages deposits and withdrawals.
                \item Emits batch events for client synchronization.
            \end{itemize}
        
        \item \textbf{Verifier.sol}
            \begin{itemize}
                \item Verifies zkSNARK/STARK proofs submitted by the off-chain prover.
                \item Enforces correctness of all rollup state transitions.
            \end{itemize}
    \end{itemize}
    
    \item \textbf{Data Availability Layer}
    This layer specifies how the transaction data is made available for anyone to reconstruct the rollup's state.
    \begin{itemize}
        \item {Mode A:} Store full transaction data in Ethereum calldata.
        \item {Mode B:} Store compressed transaction data (delta encoding, bit-packing).
        \item {Mode C:} Store data off-chain (IPFS \cite{protocol_ipfs} or Celestia \cite{celestia_how } like DA layers) with on-chain pointers.
    \end{itemize}
\end{itemize}

\subsubsection{Prototype Implementation Steps}

The development of the prototype will follow a layered approach, starting from the user-facing layer, moving through the off-chain execution pipeline, and finally integrating with on-chain contracts and the data availability layer. This phased structure ensures iterative validation and controlled experimentation.

The key implementation steps are as follows:

\begin{itemize}

\item \textbf{User Layer Development:} The first step will be to integrate with existing browser-based wallets (e.g., MetaMask) to allow users to initiate deposits, transfers, and withdrawals. In parallel, a synthetic workload generator will be built to simulate benchmark workloads for stress testing and performance evaluation.

\item \textbf{Off-chain Sequencer and Executor:} A minimal sequencer will be implemented in Node.js/TypeScript to handle the transaction pool, batching engine, and (optionally) custom scheduling policies (FIFO, priority-based). The executor will be developed alongside, featuring a Sparse Merkle Tree (SMT)-based state manager, a state transition engine for applying transactions, and a block constructor to produce ordered blocks with updated state roots.

\item \textbf{Prover Integration:} Circuits will be designed to verify token balance updates and state transitions ($\text{OldRoot} \rightarrow \text{NewRoot}$) \cite{sheybani2025zero}. The initial backend will use Circom with Groth16/Plonk for rapid prototyping \cite{iden3_circom, sui_groth16, polygon_plonky2}, while optional backends such as Plonky2 and Halo2 \cite{polygon_plonky2, zcash_halo2} will be explored for performance comparisons. Proof generation will then be integrated with the sequencer and executor pipeline.

\item \textbf{Submitter and On-chain Contracts:} A submitter component will be developed to package transactions, state roots, and proofs, and broadcast them to Ethereum. On-chain contracts, namely \texttt{RollupBridge.sol} and \texttt{Verifier.sol}, will serve as the canonical anchors by storing state roots, managing deposits/withdrawals, emitting synchronization events, and verifying zkSNARK/STARK proofs.

\item \textbf{Data Availability Layer:} Different DA strategies will be integrated and tested: storing full transaction data in Ethereum calldata (Mode A), compressed calldata (Mode B), and off-chain storage solutions such as IPFS \cite{protocol_ipfs} or Celestia \cite{celestia_how} with on-chain commitments (Mode C).

\item \textbf{Benchmarking and Experimental Harness:} Once the pipeline is functional, benchmarking scripts will be developed to drive synthetic workloads (modeled using a Poisson process) and collect metrics such as throughput (tx/sec), latency, gas cost, storage footprint, and DA efficiency. This harness will allow systematic experimentation across batch sizes, timeouts, scheduling policies, proof backends, and DA modes.

\item \textbf{Modular and Extensible Framework:} Throughout implementation, the system will be designed as a modular framework, drawing inspiration from zkSync and Polygon zkEVM, while avoiding direct code reuse. This ensures transparency and control in experimentation, while enabling future extensions to richer transaction types (ERC20, ERC721 \cite{vogelsteller2015erc20, vogelsteller2018erc721}), multi-sequencer architectures, and zkEVM circuits for general-purpose contract execution.

\end{itemize}



\subsection{Experimental Design for Optimization Strategies}
\subsubsection{Objectives}

Our goal is to quantify how design choices across the \emph{sequencer, executor, prover, submitter, and data-availability (DA)} layers shape end-to-end performance of a modular ZK–Rollup. We isolate parameters and run controlled sweeps to expose trade-offs in throughput, latency, and cost (with reproducible instrumentation).

\begin{itemize}
  \item \textbf{Sequencer layer (batching \& scheduling).}
  Evaluate the effect of \emph{batch size} and \emph{batch frequency} (time-/size-triggered) and \emph{scheduling policies} (e.g., FIFO vs.\ fee-priority) on
  TPS, confirmation-latency percentiles (P50/P95), tail behavior under contention, and fairness across flows.

  \item \textbf{Executor / state-update layer.}
  Assess state-update efficiency under different \emph{state-tree / commitment} options and \emph{block construction} strategies (e.g., miniblocks vs.\ macro-batches), measuring execution time per block, commit cadence, and bytes committed.

  \item \textbf{Data availability strategies.}
  Compare Mode~A: \emph{raw calldata}, Mode~B: \emph{compressed/state-diff calldata}, and Mode~C: \emph{off-chain DA with on-chain commitments} (e.g., DAC/DAL) in terms of DA bytes, gas/tx, and end-to-end latency; include external-DA options (e.g., content-addressed storage, modular DA layers) \cite{protocol_ipfs,celestia_how}.

  \item \textbf{Proof-system backends.}
  Contrast \texttt{Groth16}, \texttt{Plonky2}, and \texttt{Halo2}—including recursion when applicable—for \emph{prover wall-clock time}, \emph{parallelism scaling}, \emph{peak memory}, and \emph{L1 verification cost (gas)} \cite{iden3_circom,sui_groth16,polygon_plonky2,zcash_halo2}.

  \item \textbf{Multi-objective trade-off mapping.}
  For each configuration, record \textbf{TPS}, \textbf{end-to-end and confirmation latency}, \textbf{gas/tx} (broken down by execution, proof verify, DA), \textbf{proof generation time}, and \textbf{DA bytes}. Fit response surfaces over \emph{batch size $\times$ scheduler $\times$ DA mode} and report Pareto frontiers to guide scalable rollup deployments.
\end{itemize}

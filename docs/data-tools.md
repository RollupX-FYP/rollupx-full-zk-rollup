# Data Tools Integration

The Data Tools pipeline consumes telemetry generated by the RollupX components to provide a statistical evaluation of the runtime experiments. This document explains how metrics are collected, structured, and analyzed.

## Metrics Generation Flow

During a benchmark run (e.g., via `run_experiment.sh`), telemetry is written to a designated metrics directory, typically `benchmark-suite/metrics/<experiment_id>/<run_id>/`. The full end-to-end flow connects the Workload Generator, Sequencer, Executor, and Submitter.

1. **Workload Generator (`benchmark-suite/workload/poisson_generator.py`)**
   - **Writes:** `workload_<exp_id>.json` and `tx_log_<run_id>.csv`
   - **Details:** The generator tracks transaction submission status, latency, and experiment metadata. A transaction mix (A, B, C) and a seeded Poisson distribution dictate the load.

2. **Executor (`executor/src/grpc.rs`)**
   - **Writes:** `executor_<exp_id>.json`
   - **Details:** The Executor operates in a `grpc_passthrough` mode during typical benchmarking. It listens for `publish_batch` gRPC calls and asynchronously persists metadata (e.g., `batch_count`, `received_tx_count`, `forwarded_batch_count`, `duration_s`) locally.
   - **Note on Mocked Data:** To remain truthful to the execution environment, the executor does *not* invent cryptographic proof timings. The `proof_generation_times_ms` array is empty, and `total_proved_txs` is recorded as 0.

3. **Submitter (`submitter/src/daemon.rs`)**
   - **Writes:** `submitter_metrics.json`
   - **Details:** The Submitter outputs a JSON-Lines file. For each batch finalized on L1 (or simulated via OffChain DA), it appends a row containing `l2_l1_latency_ms`, `gas_saved`, `compression_ratio`, `gas_used_per_batch`, and `da_mode`.

4. **Experiment Orchestrator (`benchmark-suite/scripts/run_experiment.sh`)**
   - **Writes:** `run_metadata.json` and `run_status.json`
   - **Details:** The bash scripts dump the full configuration snapshot (including hardware, Rust/Python versions, and git commits) and track whether the run passed or failed.

## Analyzing Results (`aggregate.py`)

The primary entry point for analysis is `data-tools/aggregate.py`.

```bash
python3 data-tools/aggregate.py --metrics_root benchmark-suite/metrics --output benchmark-suite/metrics/all_results.csv --include_failed
```

**Aggregation Logic:**
1. It crawls the `metrics_root` looking for `run_status.json`.
2. It attempts to load `workload_<exp_id>.json`, `executor_<exp_id>.json`, and `submitter_metrics.json` for each run.
3. **Fallback Path:** If the executor is running as a daemon and logging outside the specific `run_id` directory, `aggregate.py` intelligently falls back to search for `executor_<exp_id>.json` in the parent experiment directory or the root `metrics/` directory.
4. It safely handles `None` or empty arrays (like the mocked executor `total_proved_txs` and `proof_generation_times_ms`) without throwing errors.
5. It merges everything into a consolidated `all_results.csv`.

## Statistical Computations (`stats.py`)

```bash
python3 data-tools/stats.py --input benchmark-suite/metrics/all_results.csv --output benchmark-suite/metrics/stats_summary.csv
```

This script reads `all_results.csv` and computes:
- Mean and Standard Deviation (±std)
- Percentiles (P50, P95, P99)
- 95% Confidence Intervals (CI95)
- Normalized deltas (Δ) vs. a baseline experiment (if available).

## Orchestration Pipeline

We provide a streamlined `run_pipeline.sh` script in the repository root. This script executes the entire data-tools pipeline end-to-end.

```bash
# Provide the target experiment ID (optional)
export METRICS_ROOT="benchmark-suite/metrics"
./run_pipeline.sh test_exp
```

**Pipeline Sequence:**
1. Aggregation (`aggregate.py`)
2. Statistics computation (`stats.py`)
3. Plot generation (`plots/*.py`):
   - Pareto frontiers (`pareto_frontier.py`)
   - Throughput bars (`throughput_bar.py`)
   - Latency CDFs (`latency_cdf.py`)
   - Jain's Fairness Index (`fairness.py`)
   - Cost heatmaps (`cost_heatmap.py`)
   - Sensitivity matrices (`sensitivity.py`)
4. Report generation (`report/generate_md.py`) -> outputs `thesis_summary.md`

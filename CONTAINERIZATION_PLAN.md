# Full-Stack Containerization Plan (RollupX)

## Objective

Containerize the full RollupX stack so it can be brought up reproducibly with Docker Compose, fix current runtime/script mismatches, and validate that all core components work together end-to-end.

## Scope

In scope:

- Hardhat L1 node + contract deployment
- Sequencer
- Executor (real RISC0 path)
- Submitter
- Benchmark workload runner
- Data/metrics persistence and validation scripts

Out of scope:

- UI implementation (`zk-rollup-ui` is still minimal/stub)
- Production hardening (TLS, auth, horizontal scaling)

## Current Gaps to Fix

1. No root `docker-compose.yml` orchestrating full runtime.
2. Missing Dockerfiles for key services (`sequencer`, `executor`).
3. Existing Dockerfiles are partial/outdated for full orchestration usage.
4. Benchmark automation mismatches:
   - legacy sequencer binary name in script (`rollup_sequencer` vs current `sequencer`)
   - `/health` probing assumption mismatches current sequencer API behavior
   - `ROLLUPX_CONFIG` generated by scripts is not consumed by current sequencer startup code.
5. No single verification workflow proving all services are healthy and integrated.

## Target Runtime Topology

Single Compose project with these services:

1. `hardhat` (L1 RPC at `8545`)
2. `contracts-deployer` (one-shot deploy, writes addresses/artifacts)
3. `executor` (gRPC at `50051`, trace/proof artifact persistence)
4. `submitter` (consumes executor stream, submits on-chain)
5. `sequencer` (JSON-RPC at `3000`)
6. `benchmark-runner` (on-demand profile for workload runs)
7. `data-tools` (on-demand profile for report pipeline)

Named volumes for persistence:

- `sequencer_db`
- `executor_state`
- `executor_traces`
- `executor_risc0`
- `submitter_data`
- `metrics_data`
- `runtime_config` (addresses/config handoff)

## Implementation Plan

## Phase 1 - Standardize Container Build Assets

Deliverables:

- `sequencer/Dockerfile` (release binary, runtime image)
- `executor/Dockerfile` (nightly `2024-08-01`, RISC0 host integration)
- update `submitter/Dockerfile` for runtime config + robust entrypoint
- update `benchmark-suite/Dockerfile` to Python 3.11+ and required deps
- optional root `.dockerignore` to reduce build context

Key decisions:

- Build in multi-stage images, run as non-root where practical.
- Keep runtime images minimal but include required system dependencies (`cmake`, `nasm`, `protobuf`, etc. where needed).

## Phase 2 - Compose Orchestration

Deliverables:

- root `docker-compose.yml` with dependency graph, healthchecks, volumes, networks
- `compose/.env.example` for configurable values
- profiles: `core`, `bench`, `report`

Startup order:

1. `hardhat`
2. `contracts-deployer`
3. `executor`
4. `submitter`
5. `sequencer`
6. benchmark/report jobs (on demand)

## Phase 3 - Fix Runtime/Script Mismatches

Code/documentation changes:

1. **Sequencer config loading**
   - support `ROLLUPX_CONFIG` env var with fallback to `config/default.toml`.
2. **Benchmark scripts**
   - fix sequencer binary default path/name.
   - replace `/health` dependency with reliable probe strategy:
     - either add explicit sequencer `/health` endpoint, or
     - probe JSON-RPC endpoint with deterministic request/response check.
3. **Address/config handoff**
   - deployment step writes bridge/verifier addresses to shared runtime file.
   - sequencer/submitter read from runtime config (or generated env) to avoid manual edits.
4. **RUN_GUIDE/USAGE alignment**
   - add Docker-first path and note native fallback path.

## Phase 4 - End-to-End Validation Harness

Deliverables:

- `scripts/verify_stack.sh` (or equivalent) to assert component readiness and integration.
- optional `scripts/smoke_benchmark.sh` for one minimal benchmark run.

Validation checks:

1. Hardhat responds to `eth_blockNumber`.
2. Contracts deploy succeeds and addresses are discoverable.
3. Sequencer accepts transactions on `:3000`.
4. Executor receives/seals/publishes batches; trace lifecycle contains:
   - `generated -> persisted -> proved -> published`
5. Submitter consumes stream and submits to L1 (tx hash + receipt path present).
6. Benchmark run writes:
   - `workload_*.json`
   - `tx_log_*.csv`
   - `run_status.json`
7. Data tools pipeline generates:
   - `all_results.csv`
   - `stats_summary.csv`
   - plot artifacts
   - summary markdown report

## Phase 5 - CI Sanity Path

Deliverables:

- CI workflow job:
  - build key images
  - `docker compose up` core stack
  - run short smoke benchmark
  - publish artifacts on failure/success

Goal:

- Prevent future drift between scripts, docs, and runtime behavior.

## Task Breakdown (Execution Order)

1. Add/fix Dockerfiles (`sequencer`, `executor`, `submitter`, `benchmark-suite`).
2. Add root Compose + env + volume wiring.
3. Implement runtime config handoff from deployer to services.
4. Patch sequencer config loading (`ROLLUPX_CONFIG` support).
5. Patch benchmark scripts (binary name, probe strategy).
6. Add stack verification scripts.
7. Add CI smoke workflow.
8. Update `RUN_GUIDE.md` + `USAGE.md` Docker sections.

## Acceptance Criteria

The plan is complete when all are true:

1. `docker compose up -d` starts core services without manual per-service edits.
2. One command can run a minimal benchmark and produce metrics artifacts.
3. Proof/trace lifecycle artifacts are present for benchmarked batches.
4. Submitter records on-chain submission activity against local Hardhat.
5. Documentation exactly matches actual commands and image behavior.
6. CI smoke test passes on a clean runner.

## Risks and Mitigations

1. **Executor build complexity / long image builds**
   - Mitigate with layer caching and pinned toolchains.
2. **Cross-service startup race conditions**
   - Mitigate with healthchecks + explicit waits + one-shot deployer completion signal.
3. **Config drift between local and container paths**
   - Mitigate with shared runtime config volume + env-based overrides.
4. **Benchmark script assumptions regressing**
   - Mitigate via CI smoke benchmark using same scripts in containers.

## Recommended Rollout Strategy

1. Land Dockerfiles + Compose skeleton first.
2. Land runtime/script fixes next.
3. Land validation scripts and CI last.
4. Keep native run path available until Docker path is stable across repeated runs.
